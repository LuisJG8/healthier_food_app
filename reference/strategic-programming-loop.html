Strategic Programming Loop

Before asking AI or yourself to write code, make the bet explicit. Great strategic programmers keep product intent, system shape, QA, and delivery flow visible at the same time.

1. Intent

What user moment are we preserving?

For BetterBite: the user wants a better version of the food they already want, not a moralized replacement.

2. Bet

What must be true?

Example: category, texture, flavor, and convenience similarity matter more than a generic health score.

3. Slice

What is the smallest proof?

Choose one narrow vertical path: scanned food, ranked alternatives, UI explanation, and feedback capture.

4. Evidence

How will we know?

Use tests, examples, product review, telemetry, and user feedback to check the riskiest assumption first.

Risk Map

Risk
Strategic question
Evidence to collect
User risk
Would a real person accept this as the same craving and occasion?
Concrete side-by-side examples, rejection reasons, saved swaps, short interviews.
Code risk
Can this change land without hiding complexity or breaking adjacent flows?
Small change list, focused tests, clear ownership boundary, easy rollback.
Data risk
Do the available product fields support the ranking claim?
Open Food Facts field audit, fallback behavior, low-confidence states.
Release risk
Can we expose the slice safely while learning from it?
Feature flag, demo dataset, observability, manual QA script, rollback plan.

Rule of thumb: if a task only says what to code, it is tactical. If it says what decision the code will clarify, it is strategic.
